# TAPD Ticket Confirmation

## 核心原则

创建或更新 TAPD 单据前，不能把不确定信息当作默认值执行。只要存在歧义，必须先让用户选择，等用户确认后再调用 TAPD 写入接口。

## 必须确认的场景

遇到以下任一情况，先用 `AskQuestion` 给出选择项，不要直接创建、更新或关联单据：

1. 用户没有明确 `workspace_id`，或存在多个可能项目，例如 `CF项目组`、`CF主题项目组`。
2. 用户没有明确要创建的对象类型，例如需求、任务、缺陷、评论、关联关系。
3. 用户没有明确需求类别、缺陷类型、优先级、严重程度、处理人、模块、版本、迭代等字段，且该字段会影响建单结果。
4. 用户描述了日志、文件夹、报错类型、主题范围、截图范围，但筛选规则可能有多种理解。
5. 用户提到“如果能找到截图”“相关日志”“那几个日志”等不完全确定的附件或证据范围。
6. 已经创建过相似单据，可能需要复用、补充、删除、重建或避免重复创建。

## 执行流程

1. 先读取用户最新要求，按最新消息覆盖旧假设。
2. 如果信息完整且没有歧义，可以直接执行。
3. 如果有歧义，先列出选择项让用户选，不要口头声明默认方案后继续执行。
4. 用户确认后，只按确认后的范围执行。
5. 创建完成后，返回单据链接，并说明哪些内容按用户确认范围处理了。

## 选择项要求

选择项要具体，优先使用已有事实，例如：

- 项目：`36787060 CF项目组`、`40629242 MT项目组`
- 范围：只创建 `FILE_PRELOAD_ERROR`、包含所有 `lua_error_message`、只处理当前打开日志
- 内容：只贴报错原文、包含日志路径、包含截图路径、上传/关联附件
- 单据关系：创建需求并关联缺陷、只创建缺陷、不创建新需求而关联已有需求

不要使用“我默认”“优先按”“先按某个项目处理”这类未确认动作。

## TAPD 写入前检查

调用 `create_story_or_task`、`create_bug`、`update_story_or_task`、`update_bug`、`create_comments`、`entity_relations` 等写入接口前，确认：

1. 项目 ID 已由用户明确给出或从选择项确认。
2. 单据类型和数量已确认。
3. 标题规则和描述内容范围已确认。
4. 是否关联需求、是否附截图、是否查重已确认。

若其中任何一项不确定，停止写入并询问用户。
