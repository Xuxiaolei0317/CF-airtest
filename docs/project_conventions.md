# 项目结构与命名约定

本文档用于约束 `CF-airtest` 的日常增量开发，目标是降低多人协作时的目录噪音和入口分歧。

## 1. 目录分层

- `CF/`、`MT/`：按游戏拆分的业务测试模块。
- `run_tests.py`：统一调度入口，默认从这里触发模块执行与 testlist 执行。
- `testlists/`：业务测试点编排清单（只描述“跑什么”，不写执行逻辑）。
- `log/`：运行期产物目录（报告、截图、失败现场等）。
- `docs/`：工程规范和操作说明文档。

## 2. 脚本命名

- 业务模块脚本使用前缀：`CF_*.py` / `MT_*.py`（例如 `CF_CashGo.py`）。
- 公共编排、框架能力、行为树工具使用小写前缀：`cf_*.py` / `mt_*.py`（例如 `cf_flow_runner.py`）。
- 新增脚本时优先复用现有前缀规则，不混入第三种风格，避免目录可读性下降。

## 3. 统一入口约定

- 正式回归、批量执行、CI 场景统一使用 `run_tests.py`。
- 业务脚本中的 `__main__` 保留为调试兜底，不作为团队默认执行方式。
- 新增执行参数应优先在 `run_tests.py` 暴露，再透传给具体业务脚本。

## 4. testlist 约定

- `testlists/*.json` 或 `testlists/*.yaml` 中每条用例都应有唯一 `id`。
- `game` 仅允许 `cf` / `mt`，`module` 与 `run_tests.py --module` 规则保持一致。
- 临时禁用项使用 `enabled: false`，不要直接删除条目，便于追踪覆盖面。

## 5. 产物与版本控制

- 运行产物（截图、dump、临时报告、pycache）不入库。
- 通过项目 `.gitignore` 统一忽略，不依赖个人全局 git 配置。
- 新增运行期目录时，同步检查 `.gitignore` 是否需要补充。
